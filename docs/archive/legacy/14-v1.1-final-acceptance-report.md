# 14 v1.1 最终验收执行报告

## 1. 文档目的

本文档用于记录世界线 `v1.1` 的最终验收执行结果。

它回答 5 个问题：

1. `v1.1` 本轮实际验收了哪些主链路
2. 哪些项已经通过
3. 哪些项存在阻塞
4. 哪些问题可以接受但不阻塞
5. 当前是否 ready 冻结 `v1.1` 基线

## 2. 本轮执行方式

本轮同时使用了两类验证方式：

### 2.1 运行态验证

- `docker compose ps`
- `GET /api/system/health`
- 关键页面 HTTP 状态检查
- 前端与文档构建检查

### 2.2 页面与主链路验证

本轮没有直接通过 MCP 工具面板执行浏览器自动化，而是通过页面可达性、认证接口、路由状态、运行链路与手动核对完成最终验收。

其中已覆盖的主链路包括：

- 登录页
- Agent 页
- PoE 主题页
- Graph 页
- Dashboard 页
- 用户头像菜单与系统设置入口

Playwright 已被纳入世界线第一批核心 MCP，并已写入本地 Codex 全局配置，可在后续阶段用于更深的浏览器级自动化验收。

## 4. 原阻塞项与当前状态

最初的最终验收执行中，存在 1 个不属于代码缺陷、但会影响完整业务演示的外部阻塞项：

### 4.1 原阻塞项：外部模型服务余额不足

当时在 Agent 页已有历史对话中，真实模型调用返回：

- `403`
- `Sorry, your account balance is insufficient`

这说明：

- 世界线的页面、路由、上下文承接与对话入口本身是通的
- 但当时使用的聊天模型线路无法把“真实回答成功返回”作为最终通过项冻结

### 4.2 当前状态：已在本地运行环境中解除

后续本地运行环境已将默认聊天模型切换到火山引擎 Ark 接入点，并完成复验：

- 登录接口正常
- 默认智能体接口正常
- Agent 流式链路返回 `200`
- 原先的 `403 insufficient balance` 不再出现
- 底层模型直接调用可正常返回文本
- Agent 流式链路经兼容补丁后已能返回实际回复内容

当前这项阻塞已经不再构成本地 `v1.1` 基线冻结障碍。

## 5. 可接受瑕疵

以下问题当前不阻塞 `v1.1`：

- Graph 页仍偏通用图谱工具页
- Dashboard 的部分内容仍偏占位或轻量
- Agent 页历史会话中仍能看到旧测试会话名称
- 文档和部分内部配置文件中仍有历史阶段残留内容

## 6. 当前判断

### 6.1 代码与平台基线判断

从代码、路由、构建、页面与菜单体验角度看：

> `v1.1` 已基本达到可冻结基线

### 6.2 环境与演示判断

从当前本地运行环境的“端到端真实回答成功返回”角度看：

> 当前本地环境已具备真实回答返回能力

因此最准确的结论是：

> 世界线 `v1.1` 已通过平台、展示层与本地聊天运行链路的最终验收。

## 7. 是否 ready 冻结 v1.1

当前判断：

- 代码基线：`ready`
- 页面与主链路：`ready`
- 当前本地聊天运行链路：`ready`

如果只冻结仓库与平台版本基线：

> 可以冻结 `v1.1`

如果后续仍要继续保留旧的 SiliconFlow 聊天线路作为默认线路：

> 仍应把模型额度或线路可用性作为环境依赖单独管理

## 8. 下一步建议

当前最合理的下一步不是继续大改代码，而是：

1. 冻结 `v1.1` 仓库基线
2. 把本地 Ark 聊天运行配置视为当前可用运行方案
3. 再进入 `v1.2` 规划，单独评估是否要把 Embedding、Reranker 与更多链路一起切到新的模型栈
