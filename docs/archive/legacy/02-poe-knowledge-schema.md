# 02 PoE 知识方案

## 目标

PoE 模块不是把游戏资料简单堆进知识库，而是要形成一套适合检索、问答、图谱和推荐共同使用的结构化知识方案。

## 一级知识分类

PoE 首期建议采用以下 7 个一级分类：

1. 职业与升华
2. 技能与辅助
3. 核心机制
4. 装备与 Build
5. 成长与推进
6. 常见问题与误区
7. 赛季与版本说明

## 内容类型

当前样本数据实际落地的内容类型包括：

- `concept_card`
- `build_card`
- `progression_card`
- `faq_card`
- `patch_note_card`
- `recommendation_card`

## Metadata 设计原则

PoE metadata 需要同时支持：

- 过滤检索
- 主题问答
- 图谱挂接
- 推荐候选筛选

因此字段应尽量包含：

- 内容类型
- 分类
- 主题
- 适用对象
- 难度与预算
- 玩法标签
- 场景标签
- 图谱引用

## 当前样本资产

当前仓库已落地：

- 首批 13 张 PoE 样本知识卡
- 3 条图谱闭环数据
- 3 条推荐候选集数据

这些数据已经足以支撑毕业设计阶段的演示与结构说明。

## 问答模板方向

PoE 模块适合以下 5 类问题模板：

1. 解释型问题
2. 对比型问题
3. 选择型问题
4. 推进型问题
5. 纠错型问题

示例：

- 这个 Build 为什么适合新手？
- 我现在打到黄图，防御不够应该先补什么？
- 死灵和锐眼分别适合什么玩家？

## 图谱建模方向

PoE 图谱首期建议围绕以下节点类型：

- Class
- Ascendancy
- SkillGem
- SupportGem
- Mechanic
- Item
- Build
- ContentGoal

典型关系包括：

- `has_ascendancy`
- `uses_skill`
- `scales_with`
- `depends_on`
- `recommended_for`
- `progresses_to`

## 推荐 MVP 方向

推荐 MVP 当前采用规则驱动方案，而不是复杂模型：

- 根据经验等级、预算、玩法偏好、内容目标进行筛选
- 输出一个推荐候选
- 挂接对应 Build 卡、知识卡和图谱闭环
- 把结果带入 Agent 对话页继续解释
