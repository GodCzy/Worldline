## 4. 页面入口与路由关系

### 4.1 路由策略

第一版建议采用如下路由：

```text
/worldline
  -> 世界线总入口 / 模块选择页
/worldline/:themeId
  -> 具体模块的世界线工作区
```

其中：

- `/worldline`
  - 负责解释世界线工作模式
  - 负责模块切换
  - 负责恢复上一次工作区状态
- `/worldline/:themeId`
  - 负责某个模块的分支生成、节点查看与后续推进

### 4.2 旧路由处理

现有原型路由：

```text
/themes/:themeId/worldline
```

后续应降级为：

- 兼容跳转入口
- 模块启动器入口

不再作为长期主入口。

## 7. 数据适配与模型生成边界

### 7.1 `poeWorldlineAdapter` 的职责

[poeWorldlineAdapter.js](D:/worldline/web/src/data/worldline/poeWorldlineAdapter.js) 负责：

- 把模块现有知识资产转成世界线可消费的背景材料
- 统一输出世界线节点与证据引用结构
- 标准化模型返回结果

它不负责：

- 页面渲染
- store 管理
- 复杂决策逻辑

### 7.2 第一版的模型生成边界

第一版允许：

- 模型根据问题生成基础分支
- 用户选择后，再触发下一层分支生成

第一版不允许：

- 无约束无限分支扩张
- 重型后端世界线工作流引擎
- 完整长期持久化的分支历史系统

## 8. SVG 分支树第一版方案

### 8.1 渲染策略

第一版推荐：

> **前端可控的轻量 SVG 分支树**

不是：

- 大型图编辑器
- 自由拖拽画布
- 动态物理布局系统

### 8.2 最小结构

世界线分支树需要至少支持：

- 根问题节点
- `2 ~ 4` 条基础分支
- 当前选中线高亮
- 风险 / 证据 / 下一步标签
- 轻量连线高亮

### 8.3 前端 vs MCP

前端负责：

- 节点与连线渲染
- 焦点态
- 当前路径高亮
- 详情联动

MCP 负责：

- 抓取证据来源
- 验证页面
- 支持资料补强

MCP 不负责：

- 运行时分支树绘制

## 9. 第一版必须有的 UI 区块

第一版工作区必须有：

1. 模块选择区
2. 问题输入区
3. 世界线主舞台
4. 当前线详情区
5. 证据轨
6. 下一步动作区

这些块必须独立清晰、直观简洁，不允许挤成一个“信息很多的后台板”。

## 10. 第一版绝不能做的重型能力

以下事项明确延后：

1. 完整动态世界线引擎
2. 正式比较器 / 切换器 / 校正器
3. 拖拽编辑
4. 自由布局图编辑器
5. 完整用户自定义模块构建器
6. 批量知识导入升级
7. 多智能体正式编排

## 11. 页面级最小验收门槛

第一版页面级 smoke / 验收门槛：

1. 用户能从左侧导航进入 `世界线`
2. 用户能在世界线中选择模块
3. 首屏先看到工作区主舞台，而不是聊天流
4. 模型能输出结构化基础分支
5. 用户选中分支后，节点详情区与证据轨联动
6. 用户能继续生成下一步世界线
7. 用户能承接到 Agent / Graph
8. 工作区与聊天页一眼可区分
