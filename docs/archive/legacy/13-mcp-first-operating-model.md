# 13 MCP-first 运行模式

## 1. 文档目的

本文档用于定义世界线的 MCP-first 工作方式。

从现在开始，MCP 不再被视为“有空再接”的辅助工具，而是世界线项目规划、验证、治理和后续企业化开发中的核心能力线之一。

本文档回答 4 个问题：

1. 为什么世界线要把 MCP 视为核心能力
2. 当前已经启用和新增的 MCP 有哪些
3. 每个阶段如何判断是否需要新增 MCP
4. 如何避免为了“看起来高级”而堆无效 MCP

## 2. 为什么 MCP 是核心能力

世界线要解决的不是单一页面问题，而是一个长期演进的企业级知识平台问题。

后续阶段会持续遇到这些重复性工作：

- 仓库与发布状态核对
- 前端主链路验收与回归
- 数据库状态检查
- 外部文档与知识源抓取
- 平台治理、问题追踪和协作留痕

如果这些工作一直靠人工切换工具、手工复制、手工核对，开发效率和验收质量都会不稳定。

因此，MCP 在世界线里的价值不是“功能补丁”，而是：

- 减少重复手工操作
- 提高验证与追踪效率
- 为后续企业化治理建立统一接口层

## 3. 当前 MCP 基线

### 3.1 当前环境里已存在的辅助型 MCP

当前 Codex 环境中已经可用的辅助型 MCP 有：

- Notion
- Linear
- Figma
- Playwright

其中对世界线当前最直接有价值的是：

- Playwright

因为它可以直接用于浏览器级主链路验证、页面截图、交互回归和最终验收。

### 3.2 本轮新增并纳入世界线核心集的 MCP

本轮开始，世界线第一批核心 MCP 固定为：

- GitHub
- PostgreSQL
- Fetch
- Playwright

它们分别承担不同职责：

当前本地 Codex 配置中，GitHub、PostgreSQL 与 Fetch 已完成接入，Playwright 也已保留在全局 MCP 集内。
通常在修改全局 Codex MCP 配置后，需要重启 Codex 桌面端或开启新会话，新的 MCP 工具入口才会完整生效。

#### GitHub

用途：

- 仓库与分支状态协作
- issue / PR / release 跟踪
- 提高后续企业化协作和可追踪性

当前接入方式：

- 通过本地 Docker 版 GitHub MCP 启动
- 使用 `gh auth token` 动态提供认证
- 默认只读模式，避免误写仓库

#### PostgreSQL

用途：

- 检查本地世界线数据库结构与内容
- 做只读查询与验收核对
- 支撑后续后台、运营、治理与数据排查

当前接入方式：

- 连接本地运行中的 Worldline PostgreSQL
- 目标地址：`127.0.0.1:5433`
- 数据库：`yuxi_know`

#### Fetch

用途：

- 抓取官方文档、公告、网页资料
- 支撑后续知识源收集、方案设计与 RAG 资料整理
- 减少人工网页复制整理成本

当前接入方式：

- 直接使用本机 Python 环境中的 `mcp_server_fetch`

#### Playwright

用途：

- 页面验收
- 路由与菜单回归
- UI 级 smoke
- 截图与真实交互验证

当前状态：

- 已在全局 Codex 配置中存在
- 本轮 `v1.1` 最终验收已直接使用它完成真实浏览器核对

## 4. MCP-first 的阶段判断规则

从现在开始，每个 substantial phase 都要问 3 个问题：

1. 当前阶段最重复、最耗时、最容易出错的工作是什么
2. 现有 MCP 是否已经足够覆盖这些工作
3. 是否存在一个新增 MCP，能明显减少这类重复工作

只有当答案明确时，才新增 MCP。

## 5. 什么时候应该安装新 MCP

满足以下任一条件，就应认真考虑在当前阶段安装：

- 该 MCP 会在本阶段被反复使用
- 该 MCP 能明显缩短人工验证时间
- 该 MCP 能提升可追踪性、可审计性或可观测性
- 该 MCP 比临时脚本更稳定、更可复用

## 6. 什么时候不应该安装新 MCP

以下情况不建议装：

- 只是“看起来高级”
- 当前阶段只会用一次
- 需要大量认证、接线、权限准备，但当前阶段根本不会高频使用
- 已有现成 MCP 或本地工具链已经足够

## 7. 与世界线阶段规划的关系

### v1.1

当前 `v1.1` 是稳定化与治理化版本，因此 MCP 应优先用于：

- UI 验收
- 数据检查
- 发布与仓库追踪
- 外部资料抓取

不应用于扩大版本范围。

### v1.2 及以后

后续阶段再考虑引入更多 MCP，用于：

- 正式知识源采集
- 数据治理
- 研发协作
- 问题跟踪
- 发布质量与运维观测

## 8. 当前建议的安装优先级

### 现在就应保持可用

1. Playwright
2. PostgreSQL
3. GitHub
4. Fetch

### 后续阶段再评估

- Sentry 类错误追踪 MCP
- Slack / 团队协作 MCP
- 更垂直的数据源或业务系统 MCP

## 9. 总结

世界线从现在开始采用 MCP-first 的思路，但不是“堆很多 MCP”，而是“让 MCP 解决真实阶段问题”。

当前第一批真正核心、且已经开始纳入项目基线的 MCP 是：

- GitHub
- PostgreSQL
- Fetch
- Playwright
