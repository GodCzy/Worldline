# 17 世界线品类定义

## 1. 为什么需要重新定义 Worldline

如果 Worldline 继续把自己定义成：

- 知识平台
- RAG 平台
- 图谱平台
- Agent 平台

它虽然合理，但不够锋利，也不够难忘。

这些词都太通用，市场上同类表达太多，很难形成真正的赛道认知。

因此，Worldline 需要一个比“知识平台”更强、更独特、更适合长期演进的品类定义。

## 2. 最终品类定义

Worldline 的正式品类定义建议为：

> **Worldline 是一套面向复杂领域的知识世界线引擎。**

更完整的对外表述是：

> **Worldline 不是一个通用问答产品，也不是一个传统知识库。  
> 它是一套把知识、证据、图谱、推荐、Agent 与治理能力组织成“可生成、可比较、可切换、可推进”的行动世界线系统。**

## 3. 世界线真正解决的问题

Worldline 不是为了回答一个孤立问题，而是为了处理这类更复杂的问题：

- 哪条路线最适合当前目标
- 在当前约束下应该走哪条线
- 这条路线的依据是什么
- 风险在哪里
- 下一步应该做什么
- 如果条件变化，是否应该切换路线

也就是说，Worldline 的最小独特单位不是一条问答，也不是一篇文档，而是：

> **一条世界线。**

## 4. 世界线由什么组成

一条世界线至少应包含：

- 当前目标
- 当前约束
- 推荐路线
- 路线依据
- 图谱位置
- 风险提示
- 下一步行动
- 可切换分支

只要这个结构成立，Worldline 就不再是一个聊天工具，而是一个路线生成与推进系统。

## 5. Worldline 不是什么

### 5.1 它不是另一个 ChatGPT

Worldline 的价值不在于做一个更强的通用聊天入口，而在于：

- 把模型能力放进知识系统
- 把对话放进主题上下文
- 把图谱和推荐变成可解释依据
- 把知识工作放进可治理平台

### 5.2 它不是一个普通知识库

普通知识库擅长：

- 存
- 查
- 看

Worldline 应该擅长：

- 选路线
- 看依据
- 换路线
- 推进下一步
- 治理路线

### 5.3 它不靠“模型更强”取胜

Worldline 的优势不是“模型比通用大模型更聪明”，而是：

- 更贴近复杂领域
- 更可控
- 更可解释
- 更适合治理与运营

## 6. Worldline 的 4 个核心动词

如果要把品类定义压缩成最关键的方法论，Worldline 应围绕这 4 个动词：

1. **生成世界线**
2. **比较世界线**
3. **切换世界线**
4. **推进世界线**

这 4 个动作才是 Worldline 长期最值得强化的“招牌能力”。

## 7. 当前仓库已经具备哪些支撑基础

当前仓库虽然还没有把“知识世界线引擎”完整做出来，但已经具备关键基础：

### 7.1 平台基础

- 主题分区
- 平台导航
- 模块入口
- 主题上下文透传
- Dashboard、Graph、Agent、系统设置等平台壳层

### 7.2 路线雏形

- 推荐候选
- 图谱闭环
- Agent 承接当前候选与闭环
- Graph 承接当前闭环

### 7.3 结构化知识基础

- 模块 taxonomy
- metadata schema
- 样本知识卡
- 图谱关系样本
- 推荐候选集

### 7.4 工程基础

- 版本冻结
- 阶段推进
- 文档体系
- `docs/context-cache/`
- MCP-first

## 8. 当前还不能硬吹的部分

为了让品类定义可信，必须明确当前还没有完整做出来的部分：

- 正式世界线生成器
- 正式世界线比较器
- 正式世界线切换器
- 正式世界线失效检测
- 生产级知识接入链路
- 图谱生产化构建

所以当前正确说法应该是：

> Worldline 已经具备“知识世界线引擎”的骨架与方向，但尚未完成全部生产化落地。

## 9. 为什么这条赛道值得占

“知识平台”这个概念太宽，也太普通。

“知识世界线引擎”则把 Worldline 的名字、结构和能力真正统一了起来：

- 为什么需要主题模块
- 为什么需要推荐
- 为什么需要图谱
- 为什么需要 Agent
- 为什么需要治理
- 为什么需要 MCP-first

它不是为了显得高级，而是为了让产品有一个真正能长期成立的方法论核心。

## 10. 主控结论

主控综合各角色结论后，给出正式判断：

> Worldline 适合占据的赛道，不是“另一个知识平台”，而是“知识世界线引擎”。  
> 接下来的版本，不应继续围绕“功能列表”推进，而应围绕“世界线生成、比较、切换、推进”这一产品母题推进。
