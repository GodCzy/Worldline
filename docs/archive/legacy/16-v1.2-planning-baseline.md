# 16 v1.2 规划基线

## 1. 文档目的

本文档用于冻结 `v1.2` 第一版的产品定位、实施边界与最小成功标准。

`v1.2` 不再被定义为“再做一个更大的主题详情页”或“在聊天页旁边多一个工作台”，而是被重新收敛为：

> **把 Worldline 做成主壳层内的顶层世界线工作区，用 AI 根据用户问题生成若干基础世界线分支，再让用户沿分支继续推进。**

这意味着 `v1.2` 的核心不是堆更多功能，而是正式建立一种新的产品工作模式：

- 保留基础聊天界面
- 新增顶层 `世界线` 工作区
- 在工作区内选择模块
- 由模型生成基础分支
- 用户选择后继续生成后续世界线
- 每个节点都可承接知识库、图谱、文档与下一步动作

## 2. 当前阶段判断

- 当前阶段：`v1.2 规划`
- 当前判断：`ready 进入 v1.2 实施准备`
- 当前最大 gap：现有原型仍偏“独立工作台页”，需要把产品入口、路由、页面结构和生成机制整体纠偏到顶层工作区模式

## 3. 本轮常驻 subagent 收敛结论

### 3.1 `system_mapper`

结论：

- 世界线下一步应从当前 `/themes/:themeId/worldline` 原型，逐步迁移成主壳层内的顶层入口。
- 最合理的壳层位置是左侧导航中：
  - `主题分区` 之后
  - `智能体` 之前
- 最合理的新路由形态是：
  - `/worldline`
  - `/worldline/:themeId`
- 当前 `/themes/:themeId/worldline` 不应继续作为长期主入口，应降级为：
  - 模块启动入口
  - 过渡跳转入口
- `v1.2` 下一步的安全主战场应保持在：
  - `web/src/layouts/AppLayout.vue`
  - `web/src/router/index.js`
  - `web/src/views/worldline/**`
  - `web/src/components/worldline/**`
  - `web/src/stores/worldlineContext.js`
  - `web/src/data/worldline/**`
  - `web/src/views/themes/ThemeDetailView.vue`

### 3.2 `product_architect`

结论：

- Worldline 的正确形态不是“一个独立出来的 PoE 工作台页面”，而是**平台级顶层工作区**。
- 正确的双界面关系是：
  - 基础聊天界面：负责连续追问、解释、深挖
  - 世界线工作区：负责展开多条分支、比较、选择、推进
- 第一版必须把“选世界线并沿线推进”的工作方式立住，而不是退化成推荐卡片页。
- 长期方向必须正式纳入：
  - 模块差异化 UI
  - 用户导入知识库 / 图谱 / 文档后生成自己的模块

### 3.3 `qa_release_auditor`

结论：

- 第一版的最小成功标准不是“界面更酷”，而是用户能在十秒内明确说出：
  - “我现在不是在普通聊天，而是在一个先看多条世界线、再沿线推进的工作区里。”
- 如果做成以下任一形态，就算方向失败：
  - 普通推荐页
  - 聊天页附属边栏
  - 后台风格配置页
- 浅色 / 暗色模式都必须强调：
  - 焦点路径
  - 分支层级
  - 当前选中线
  - 证据区与下一步动作区的分层

## 4. v1.2 第一版的准确定义

`v1.2` 第一版的准确定义是：

> **Worldline 顶层工作区第一版。**

它不是：

- 新版主题详情页
- 更复杂的推荐页
- 聊天页上方多几张卡片
- 全量动态世界线引擎
- 用户自定义模块构建器完整版

它是：

> **一个位于主壳层顶层导航中的世界线工作区。用户先选择模块，再输入问题，由模型生成若干基础世界线分支；用户选中某条线后继续展开后续分支，并在节点中查看基于知识库、图谱和文档生成的内容，然后决定下一步。**

## 5. v1.2 第一版的核心目标

`v1.2` 第一版只聚焦 5 个核心目标：

1. **建立顶层工作区入口**
   - 在主壳层左侧导航中新增 `世界线`
   - 让它成为平台级能力，而不是某个模块的附属页

2. **建立双界面模型**
   - 保留基础聊天界面
   - 新增世界线工作区
   - 两者可以互相承接，但不能互相吞并

3. **建立模块内世界线生成机制**
   - 用户先在工作区中选择模块
   - 对所选模块提出问题
   - 模型生成 `2 ~ 4` 条基础世界线分支
   - 用户选中某条线后，再继续生成下一层世界线

4. **建立节点级知识承接**
   - 节点可查看：
     - 当前线的含义
     - 这条线为何成立
     - 对应知识卡 / 图谱 / 文档证据
     - 下一步动作

5. **建立世界线感而不是推荐感**
   - UI 要让人感到：
     - 在看“分叉的未来路径”
     - 而不是“几条推荐建议”

## 6. v1.2 第一版纳入范围

### 6.1 纳入范围

#### A. 顶层世界线入口

- 左侧导航新增一级入口：`世界线`
- 进入 `/worldline`

#### B. 模块选择

- 在世界线工作区内完成模块选择
- 第一版至少支持：
  - `PoE`
- 其他模块保留占位与扩展边界

#### C. 模型生成基础世界线

- 用户输入问题 / 目标 / 约束
- 模型输出结构化世界线 JSON
- 至少生成 `2 ~ 4` 条基础分支

#### D. 节点详情与证据承接

- 点击节点后查看：
  - 路线说明
  - 风险 / 代价
  - 适用场景
  - 知识证据
  - 下一步动作

#### E. 节点推进

- 用户选中某条线后，可继续生成下一步世界线
- 用户可带着当前线承接到：
  - Agent
  - Graph
  - 模块文档

#### F. 模块差异化 UI 的边界预留

- 工作区核心机制统一
- 不同模块可有不同视觉语法、节点语义和证据表现

### 6.2 不纳入范围

以下事项明确延后，不进入 `v1.2` 第一版实施范围：

1. 完整动态世界线引擎
2. 正式世界线比较器 / 切换器 / 校正器
3. 拖拽式世界线编辑器
4. 重型图引擎或自由布局编辑器
5. 批量正式知识导入流水线升级
6. 多智能体正式编排
7. 用户自定义模块构建器完整版
8. 模型栈全量迁移

## 7. MCP-first 在 v1.2 的角色

`v1.2` 中，MCP-first 继续作为核心能力线，但职责要更清晰：

- `Playwright`
  - 浏览器级验收
  - 世界线工作区浅色 / 暗色模式回归
  - 节点交互与承接 smoke

- `Fetch`
  - 抽取外部资料
  - 补强世界线节点证据层

- `PostgreSQL`
  - 检查运行状态、配置和数据一致性

- `GitHub`
  - 规划、版本治理、验收清单与 release 协作

第一版不要求引入“绘画 MCP”做运行时主渲染。
运行时主分支图必须由前端可控渲染；后续如需图像 / 视觉生成 MCP，可用于：

- 世界线封面素材
- 模块视觉资产
- 工作区氛围图辅助

而不是替代工作区运行时 UI。

## 8. 第一版的最小成功标准

只有满足以下条件，`v1.2` 第一版才算成立：

1. 用户能从主壳层顶层入口进入 `世界线`
2. 用户进入后先看到的是模块选择与分支工作区，而不是聊天流
3. 模型能基于问题生成结构化基础分支，而不是展示固定静态推荐文案
4. 用户能一眼看出不同世界线之间的差异
5. 点击节点后能看到证据与下一步动作
6. 用户能带着当前世界线继续进入 Agent / Graph
7. 用户能明显感知：
   - 世界线工作区和基础聊天界面是两种不同工作模式

## 9. 主控结论

本轮纠偏后，`v1.2` 第一版的方向已经足够清楚：

> Worldline 的下一步，不是继续把 `/themes/:themeId/worldline` 打磨成更复杂的单页原型，而是把 `世界线` 正式提升为主壳层中的顶层工作区，再在其中完成模块选择、模型生成分支、节点证据查看与后续推进。

这条新方向已经：

- 与当前仓库能力匹配
- 与用户最新产品意图一致
- 与 `Worldline` 这个名字真正匹配

当前判断：`ready 进入 v1.2 实施准备下一轮`
