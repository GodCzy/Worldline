# 11 v1.1 规划基线

## 1. 文档定位

本文档用于冻结世界线企业化 `v1.1` 的正式规划结论。

`v1.1` 不是能力爆发版本，也不是底座重构版本，而是从当前 `v1.0.0` 基线继续前进的第一个企业化治理版本。它的作用是把现有仓库从“可运行、可展示的产品基线”提升为“可持续迭代、可发布、可验证的版本基线”。

本文档同时承担 4 个作用：

- 明确 `v1.1` 的目标、范围、非目标和实施顺序
- 冻结 `v1.1` 可以改动的安全边界与不应触碰的高风险核心层
- 定义 `v1.1` 的验收标准、回归重点和发布门槛
- 作为 `v1.1` 实施阶段的唯一上位规划依据

## 2. 版本定位

### 2.1 版本定义

`v1.1` 的定义是：

> 平台稳定化与治理化版本

它要解决的问题不是“功能还不够多”，而是“当前产品虽然可运行，但还没有形成足够清晰、稳定、可验证的企业化迭代基线”。

### 2.2 版本目标

`v1.1` 目标收敛为以下 4 项：

1. 稳定平台主链路  
   确保首页、登录、主题页、Agent、Graph、Dashboard 这些主路径稳定可进入、可操作、可理解。

2. 清理展示层可见问题  
   优先清理用户可见的乱码、旧文案、混乱提示、不直观引导、输入区与说明区职责不清等问题。

3. 固化企业化迭代基础  
   补齐页面引导、回归清单、阶段缓存、版本边界、验收标准与发布前检查清单。

4. 保持架构边界稳定  
   不在 `v1.1` 中提前进入底座重构，不破坏平台层与模块层分离。

## 3. 当前阶段判断

### 3.1 当前项目阶段

当前项目已正式进入：

> `v1.1` 规划阶段

### 3.2 下一阶段

下一阶段应为：

> `v1.1` 实施阶段

### 3.3 是否已经 ready 进入下一阶段

判断结论：

> 已基本 ready，可以进入 `v1.1` 实施阶段

原因是本轮三类前置收敛已经完成：

- `system_mapper` 已完成安全改动边界与高风险核心层确认
- `product_architect` 已完成目标、范围、非目标、模块拆分与实施顺序收敛
- `qa_release_auditor` 已完成验收标准、回归重点、主要风险与发布门槛收敛

### 3.4 进入实施阶段前仍需保持的硬约束

虽然已经 ready，但进入实施阶段时仍必须遵守：

- 不扩大版本范围
- 不提前进入 `v1.2` 的知识导入和深联动任务
- 不触碰高风险核心层
- 每一轮实现都必须以“可验证、可回归、可回滚”为前提

## 4. 范围

### 4.1 在 `v1.1` 范围内的内容

#### 平台展示层

- 首页
- 登录页
- 主题分区页
- PoE 主题页
- Agent 页
- Graph 页
- 左侧导航 / 顶部导航
- 用户可见帮助文案、引导文案与提示语

#### 平台交互层

- 主题上下文展示清晰化
- 推荐上下文与图谱闭环上下文的表达优化
- 输入区、说明区、操作区职责清晰化
- 页面内引导与基础提示补齐
- Agent 页和 Graph 页的主链路体验稳定化

#### 工程与交付层

- 正式的 `v1.1` 规划文档
- `docs/context-cache/` 阶段缓存沉淀
- `v1.1` smoke checklist
- `v1.1` 人工回归清单
- `v1.1` 验收标准与发布前检查项

### 4.2 适合改动的低风险区域

- `docs/**`
- `README.md`
- `web/src/views/**`
- `web/src/components/**`
- `web/src/layouts/**`
- `web/src/router/**`
- `web/src/stores/**` 中纯前端上下文与显示逻辑部分
- `src/config/static/info.template.yaml`
- `data/poe/**`

### 4.3 可谨慎进入的中风险区域

只有在实现阶段确有必要时，才可谨慎进入：

- `web/src/views/AgentView.vue`
- `web/src/views/GraphView.vue`
- `server/routers/system_router.py`
- `server/routers/dashboard_router.py`
- `docker-compose.yml`
- `.env.template`
- 运行与部署相关外壳文件

## 5. 非目标

`v1.1` 明确不做以下事项：

- 不重构 `src/knowledge/**`
- 不重构 `src/agents/common/**`
- 不改 `src/services/chat_stream_service.py`
- 不启动正式 PoE 大规模知识导入
- 不做推荐系统算法升级
- 不做图谱生产化改造
- 不做多租户
- 不做权限体系重构
- 不做多智能体正式编排落地
- 不做数据库结构大迁移
- 不做移动端或复杂 BI 扩展

一句话概括：

> `v1.1` 是稳定化与治理化版本，不是功能扩张版本。

## 6. 高风险核心层

以下区域在 `v1.1` 第一轮实施中视为红区：

- `src/knowledge/**`
- `src/agents/**`
- `src/agents/common/**`
- `src/services/chat_stream_service.py`
- `src/services/agent_run_service.py`
- `src/services/run_worker.py`
- `src/services/run_queue_service.py`
- `src/storage/**`
- `src/repositories/**`
- `server/utils/lifespan.py`
- `server/main.py`

这些文件一旦被修改，通常会同时影响：

- 鉴权
- 会话
- RAG
- 图谱
- 队列
- 数据库初始化
- 流式输出
- 运行时稳定性

因此它们不应成为 `v1.1` 的主落点。

## 7. 模块拆分

建议将 `v1.1` 拆成 5 个工作模块。

### 模块 A：平台可见层治理

- 品牌一致性
- 文案一致性
- 乱码清理
- 历史残留口径清理

### 模块 B：主链路交互收口

- 首页到主题页
- 主题页到 Agent
- 主题页到 Graph
- 登录后进入默认智能体
- Agent 输入区、说明区、推荐区、图谱区逻辑清晰化

### 模块 C：PoE 模块入口体验优化

- 当前推荐、当前闭环、建议提问的表达优化
- 让 PoE 模块入口更像产品入口，而不是样本页

### 模块 D：回归与验收体系

- `v1.1` smoke checklist
- `v1.1` 手工验收路径
- 必须通过项、可接受瑕疵项、阻塞项

### 模块 E：文档与阶段治理

- 正式规划文档
- 缓存文档
- 阶段判断
- 进入 `v1.2` 的前置条件定义

## 8. 实施顺序

建议严格按下面顺序推进：

1. 冻结 `v1.1` 规划基线  
   即当前文档与阶段缓存。

2. 先做展示层主链路  
   优先收首页、登录、主题页、Agent、Graph 的用户可见问题。

3. 再做上下文与引导层  
   把主题上下文、推荐上下文、图谱闭环上下文的页面表达统一起来。

4. 再做回归清单与验收基线  
   形成稳定的 smoke 路径与手工回归路径。

5. 最后进入发布收口  
   用 QA 视角统一验证并判断是否达到 `v1.1` 发布门槛。

## 9. subagent 分工与依赖

### 9.1 本轮三类常驻 subagent 产出

#### `system_mapper`

负责：

- 前端入口复核
- 后端入口复核
- 核心服务边界识别
- 高风险核心层识别
- 安全写入面划分

本轮结论：

- `v1.1` 的主安全写入面在 `docs/**`、`web/src/views/**`、`web/src/components/**`、`web/src/layouts/**`、`web/src/router/**`、部分前端 store 与静态配置
- 当前不应进入 `src/knowledge/**`、`src/agents/common/**`、`src/services/*核心运行链`

#### `product_architect`

负责：

- 版本目标
- 范围
- 非目标
- 模块拆分
- 实施顺序
- 依赖关系

本轮结论：

- `v1.1` 应定义为“稳定化与治理化版本”
- 主工作面以前端展示层、平台主链路和交互清晰化为主

#### `qa_release_auditor`

负责：

- 验收标准
- 回归重点
- 主要风险
- 发布门槛

本轮结论：

- `v1.1` 的发布 gate 应由构建通过、运行通过、主链路 smoke 通过、人工回归通过 4 部分构成
- 登录、默认智能体路由、Agent 输入区可见性、主题上下文透传、PoE 推荐与图谱入口是最高优先回归面

### 9.2 进入实施阶段后的推荐协作波次

第一波实施优先：

- `frontend_worker`
- `qa_release_auditor`

如确有必要再引入：

- `backend_worker`

不建议在 `v1.1` 中优先启动：

- `knowledge_rag_engineer`
- `graph_engineer`
- `agent_architect`

除非出现“低风险外壳无法完成目标，且必须补少量后端接口或上下文能力”的情形。

## 10. 验收标准

### 10.1 自动化可验证部分

- 前端构建通过：`pnpm build`
- 文档构建通过：`npm run docs:build`
- Docker 启动通过：`docker compose up -d --build`
- 服务状态正常：`docker compose ps`
- HTTP smoke checks 通过：
  - `/api/system/health`
  - `/`
  - `/themes`
  - `/themes/poe`
  - `/agent/ChatbotAgent`
  - `/graph`
  - `/dashboard`
  - `/api/chat/default_agent`
- PoE 静态资产存在性检查通过：
  - manifest
  - recommendation candidates
  - graph loops

### 10.2 人工必须回归部分

- 登录流程与默认智能体路由
- 首页、主题页、Agent、Graph、Dashboard 主链路
- Agent 输入区是否首屏可见
- 推荐区、上下文区、输入区是否职责清晰
- Graph 当前闭环提示与默认关键词是否合理
- 文案、品牌、帮助入口是否统一且可理解
- 侧栏、三点菜单、导航与角色可见性是否直观

## 11. 发布门槛

`v1.1` 进入发布判断前，必须满足：

1. 不触碰高风险核心层
2. 不扩大版本范围
3. 完成前端构建、文档构建、本地 Docker 启动、HTTP smoke checks、主链路人工回归
4. 明确记录已知问题，尤其是：
   - 外部模型余额 / 权限 / 网络问题
   - 尚未自动化覆盖的关键链路
   - 不阻塞发布的展示层瑕疵

## 12. 主要风险

### 12.1 版本范围失控

最大风险不是做不动，而是把 `v1.2+` 的知识接入、推荐增强、图谱生产化任务混进 `v1.1`。

### 12.2 外部依赖风险

即使页面与接口链路正常，真实模型回答仍可能受第三方模型服务余额、权限和网络影响。

### 12.3 前端回归风险

主题静态资产、路由兜底、上下文透传、页面布局之间耦合较高，容易发生“修 A 退 B”的问题。

### 12.4 自动化覆盖不足

当前仍有不少关键体验依赖人工回归，因此必须先形成稳定 checklist，不能只凭单次观察判断版本可发布。

## 13. 结论

本轮三类 subagent 结果已经一致收敛到同一个版本判断：

- 当前阶段：`v1.1` 规划阶段
- 当前状态：规划已冻结
- 阶段判断：已 ready 进入 `v1.1` 实施阶段
- 实施重点：平台展示层、交互清晰化、主链路稳定化、回归与发布基线
- 最大约束：不进入高风险核心层，不扩大版本范围

`v1.1` 完成之后，才应正式进入 `v1.2` 的知识接入与检索闭环阶段。
