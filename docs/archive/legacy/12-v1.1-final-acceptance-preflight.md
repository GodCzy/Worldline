# v1.1 最终验收前整理

## 1. 文档目的

本文档用于冻结世界线 `v1.1` 在进入“最终验收执行”之前的整理结论。

它回答 5 个问题：

1. 当前 `v1.1` 到了什么程度
2. 哪些主链路已经达标
3. 哪些问题会阻塞 `v1.1` 验收
4. 哪些问题可以接受但不必阻塞
5. 哪些事项应明确延后到 `v1.2`

本文档不承担功能设计职责，也不扩张版本范围。
它的作用是把 `v1.1` 从“持续收口状态”推进到“可执行最终验收状态”。

## 2. 当前阶段判断

- 当前阶段：`v1.1` 实施阶段
- 当前子阶段：`v1.1` 最终验收前整理
- 当前判断：`已 ready 进入 v1.1 最终验收执行`

做出这一判断的依据是：

- 主链路页面均可访问
- 前端构建通过
- 文档构建通过
- Docker 运行状态正常
- 展示层第三波收口后，用户可见文案、菜单、提示和页面理解成本已经明显下降
- 常驻 subagents 的边界判断、QA 判断和展示层扫描结论一致，没有出现新的结构性阻塞

## 3. 本轮 subagent 汇总

### 3.1 `system_mapper`

职责：

- 复核最终验收阶段仍允许变动的文件边界
- 标记最终阶段绝对不能再触碰的区域
- 识别最后阶段最容易误入的高风险点

结论：

- `v1.1` 最终阶段仍允许改动的，只剩最后一层低风险前端展示文件：
  - `web/src/views/HomeView.vue`
  - `web/src/views/LoginView.vue`
  - `web/src/views/themes/ThemeHubView.vue`
  - `web/src/views/themes/ThemeDetailView.vue`
  - `web/src/layouts/AppLayout.vue`
  - `web/src/components/UserInfoComponent.vue`
  - `web/src/views/AgentView.vue`
  - `web/src/views/GraphView.vue`
  - `web/src/views/DashboardView.vue`
- 允许动的类型，只能是：
  - 文案
  - tooltip
  - placeholder
  - 空态
  - 首屏说明
  - 壳层轻布局
- 绝对不能再动：
  - `src/knowledge/**`
  - `src/agents/common/**`
  - `src/services/chat_stream_service.py`
  - `AgentChatComponent`
  - `GraphCanvas`
  - 登录流程、themeContext 协议、图谱上传/查询逻辑、配置切换逻辑

### 3.2 `qa_release_auditor`

职责：

- 输出最终验收版 smoke checklist
- 输出人工回归清单
- 定义阻塞项与可接受瑕疵
- 给出应该延后到 `v1.2` 的事项

结论：

- `v1.1` 的最终验收应按以下主链路执行：
  - 首页
  - 登录页
  - 主题分区页
  - PoE 主题页
  - Agent 页
  - Graph 页
  - Dashboard 页
  - 用户头像菜单与系统设置入口
- 会阻塞 `v1.1` 的问题只有 8 类：
  - 白屏/红色报错遮罩
  - 登录失败或异常路由
  - PoE 主题页无法进入 Agent 或 Graph
  - Agent 首屏看不到输入区或无法输入
  - Graph 主操作区失效
  - 用户头像菜单无法展开或系统设置无反应
  - 关键标题/按钮/说明存在明显乱码
  - 导航失效
- 可接受瑕疵包括：
  - 个别文案不够产品化但仍可理解
  - 少量样式不统一
  - Dashboard 某些模块偏占位
  - Graph 页仍偏通用工具界面
  - PoE 页面仍以样本数据展示为主

### 3.3 `frontend_worker`

职责：

- 进行最终一轮只读扫描
- 找出仍残留的少量低风险旧文案或提示问题
- 判断哪些页面已经足够进入 `v1.1` 基线

结论：

- 已足够进入 `v1.1` 基线的页面：
  - 首页
  - Agent 页
  - Dashboard 页
  - 主题分区页
  - PoE 主题页
- 仍有少量低风险遗留的地方主要在：
  - `GraphView.vue` 的技术术语偏多
  - “文档中心”口径与“查看文档 / 项目文档”仍可进一步统一
  - Agent 首屏信息密度略高，但不再阻塞使用
- 如果一定要做最后一个小修，最值得修的是：
  - `GraphView.vue` 的技术化术语
  - 全站文档入口口径统一

## 5. 阻塞项判断

当前没有发现必须马上修复、否则就不能进入 `v1.1` 最终验收执行的新增阻塞项。

换句话说：

> 目前 `v1.1` 已经满足“进入最终验收执行”的前置条件。

但下面这些问题如果在人工最终验收时被重新发现，应立即视为阻塞并打回：

1. 关键页面出现白屏、红色运行时报错遮罩或主操作区失效
2. 登录失败或登录后进入异常页面
3. PoE 主题页无法进入 Agent 或 Graph
4. Agent 页首屏看不到输入区或输入失效
5. 用户头像菜单或系统设置再次出现无反应
6. 关键标题、按钮、说明存在明显乱码，导致普通用户无法理解
7. 导航不可用，导致主链路切换失效

## 6. 可接受瑕疵

以下问题可以记录，但不必阻塞 `v1.1`：

1. 个别页面文案还不够像最终商业产品
2. 个别页面仍有少量技术术语
3. Dashboard 某些内容偏轻或偏占位
4. Graph 页仍然更像通用图谱工具页，而不是垂直定制业务页
5. PoE 模块当前仍以样本知识资产为主
6. 系统设置页内部内容偏轻，只要入口可用即可
7. 外部模型偶发慢响应，但平台结构与页面交互正常

## 7. 明确延后到 v1.2 的事项

以下事项不应再塞进 `v1.1`：

1. PoE 正式知识导入与批量资产接入
2. RAG 检索质量验证与优化
3. 图谱生产化构建与更深联动
4. 推荐逻辑从样本候选升级为更完整的规则执行
5. 系统设置内部完整治理能力
6. 更系统的自动化测试体系
7. 更深层的信息架构与品牌精修
8. 更完整的后台数据面板和运营指标体系

## 8. 最终验收执行建议

建议按以下顺序执行最终验收：

### 第一步：运行态确认

- `docker compose ps`
- `pnpm build`
- `npm run docs:build`
- 健康检查 `GET /api/system/health`

### 第二步：最小 smoke

依次访问并确认可用：

1. `/`
2. `/login`
3. `/themes`
4. `/themes/poe`
5. `/agent/ChatbotAgent`
6. `/graph`
7. `/dashboard`

### 第三步：人工回归

重点看 4 个点：

1. 用户能否一眼理解主要操作入口
2. 页面是否还有明显乱码
3. 导航、菜单、设置入口是否稳定
4. Agent 与 Graph 是否还能承接主题上下文

### 第四步：验收判断

如果 smoke 全通过、人工回归没有阻塞项，则：

> `v1.1` 可以进入最终验收通过结论

如果仍发现阻塞项，则：

> 只允许做最后一轮低风险展示层修复，不允许扩展到逻辑层
