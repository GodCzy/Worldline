# 18 v1.2 世界线工作区蓝图

## 1. 这次纠偏解决什么问题

此前对“世界线界面”的理解仍然偏向：

- 路线卡片页
- 主题详情页加强版
- 一个独立工作台页面

这都不够准确。

用户最新意图更明确：

- 基础聊天界面继续保留
- 世界线要进入主壳层左侧导航，成为一级工作区
- 进入世界线后，再选择不同模块
- 模型根据用户问题生成世界线分支
- 用户选择某个节点后，系统再继续生成后续世界线
- 节点中可查看知识库 / 图谱 / 文档支撑的内容
- 长期支持用户导入知识资产生成自己的模块

因此，`v1.2` 第一版要做的不是“一个独立工作台页”，而是：

> **Worldline 顶层工作区 / 世界线工作模式。**

## 2. 一句话定义

> **世界线工作区是一套把用户问题展开成多条可视化未来路径，并允许用户沿路径选择、比较、查看证据、继续生成和进入后续动作的顶层工作空间。**

## 3. 双界面模型

### 3.1 基础聊天界面

基础聊天界面继续保留。

它负责：

- 低门槛问答
- 连续追问
- 深入解释
- 对当前世界线的补充追问

### 3.2 世界线工作区

世界线工作区负责：

- 输入问题
- 生成多条基础世界线
- 可视化显示世界线分支
- 选择当前世界线
- 查看节点级知识与证据
- 继续生成下一步世界线
- 承接到 Agent / Graph / 文档

### 3.3 两者关系

两者不是替代关系，而是分工关系：

- 聊天界面是“解释层”
- 世界线工作区是“路径层”

用户可以：

- 先在世界线工作区选定一条线，再进入聊天深挖
- 先在聊天中提出问题，再回到世界线工作区查看分支结构

## 4. 顶层 IA

### 4.1 主壳层导航

推荐壳层导航顺序：

- 首页
- 主题分区
- 世界线
- 智能体
- 知识图谱
- 知识库
- 运营看板

`世界线` 应放在：

- `主题分区` 之后
- `智能体` 之前

因为它既不是单纯内容入口，也不是后台工具，而是新的主工作模式。

### 4.2 路由形态

推荐第一版路由关系：

- `/worldline`
  - 世界线总入口
  - 用于模块选择、状态恢复、工作模式介绍
- `/worldline/:themeId`
  - 某模块的世界线工作区
  - 第一版至少支持 `/worldline/poe`

旧路由：

- `/themes/:themeId/worldline`

应降级为：

- 模块启动入口
- 兼容跳转入口

而不再作为长期主入口。

## 5. 世界线工作区的首屏结构

第一版首屏必须一眼让用户知道自己在“选世界线”，不是在聊天。

推荐首屏结构：

1. **工作区头部**
   - 当前模块
   - 当前问题
   - 当前生成状态
   - 浅色 / 暗色下都清晰的舞台头图或结构背景

2. **问题输入区**
   - 用户输入问题 / 目标 / 约束
   - 触发“生成基础世界线”

3. **模块选择区**
   - 当前模块 selector
   - 第一版至少有 `PoE`

4. **世界线主舞台**
   - 根问题
   - `2 ~ 4` 条基础分支
   - 当前焦点线
   - 清晰的分叉关系与路径感

5. **节点详情区**
   - 当前分支解释
   - 适用对象
   - 风险 / 代价
   - 这条线为什么成立

6. **证据轨**
   - 对应知识卡
   - 对应图谱闭环
   - 对应文档或资料

7. **下一步动作区**
   - 沿当前线继续生成
   - 进入对话
   - 查看图谱
   - 查看模块文档

## 6. 模型生成逻辑的正确边界

用户这次明确要求：

- 世界线不是依靠静态写死文案去凑出来
- 而是由模型根据用户问题生成基础分支
- 用户选择后，再继续生成后续世界线

因此，第一版必须正式纳入：

### 6.1 模型生成的内容

模型至少要生成：

- 根问题摘要
- `2 ~ 4` 条基础世界线
- 每条线的：
  - 标题
  - 简述
  - 适用对象
  - 风险 / 代价
  - 推荐下一步
  - 证据引用键

### 6.2 前端负责的内容

前端负责：

- 世界线舞台布局
- 分支树渲染
- 焦点反馈
- 节点详情面板
- 证据轨
- 下一步动作区

### 6.3 MCP 的位置

MCP 负责增强，而不是承担运行时绘制：

- `Fetch`：补外部资料
- `Playwright`：浏览器级验收
- `PostgreSQL`：状态 / 数据检查
- `GitHub`：规划 / release / issue

后续若需要图像或视觉生成类 MCP，可以用于：

- 模块视觉素材
- 场景图
- 氛围图

但**运行时世界线树仍应由前端控制渲染**。

## 7. 模块差异化 UI

第一版必须正式承认：

> 世界线的核心机制统一，但不同模块可以有不同 UI 语法。

这意味着：

- 平台层统一：
  - 问题
  - 分支
  - 当前线
  - 证据
  - 下一步
- 模块层可变化：
  - 节点样式
  - 风险表达
  - 标签体系
  - 证据展示方式
  - 视觉语言

例如：

- PoE 模块像 build 分支 / 成长路径
- 企业 SOP 模块像流程分叉 / 处理策略
- 学习模块像学习路径 / 进阶分支

## 8. 用户自定义模块的长期方向

这不是 `v1.2` 第一版要做完的东西，但必须作为长期方向正式写入：

未来用户可：

- 导入知识库
- 导入图谱
- 导入文档
- 基于基础世界线模板
- 创建自己的世界线模块

也就是说，Worldline 的长期方向不是“内容平台”，而是：

> **世界线模块生成与管理平台。**

## 9. 第一版必须做什么

### 必须做

- 顶层 `世界线` 入口
- 模块选择
- 基础问题输入
- AI 生成 `2 ~ 4` 条基础世界线
- 分支树式展示
- 节点详情与证据轨
- 用户选中后继续生成下一步世界线
- 承接到 Agent / Graph / 文档

### 可以做，但不是第一优先

- 更复杂的节点比较
- 更强的分支切换动画
- 更丰富的模块皮肤
- 更戏剧化的视觉动效

### 明确不做

- 重型动态图编辑器
- 拖拽式工作流系统
- 完整世界线比较器 / 校正器
- 完整自定义模块构建器
- 多智能体正式编排

## 10. 第一版的最小成功标准

只有满足以下条件，第一版才算站住：

1. 用户一眼知道这是“世界线工作区”，不是聊天页
2. 用户能看到结构化分支，而不是推荐卡片列表
3. 用户能选一条线并沿线继续推进
4. 用户能看到为什么是这条线
5. 用户能从当前线进入 Agent / Graph / 文档
6. 用户能理解：
   - 这是一个新的工作模式
   - 不是普通问答的附属功能

## 11. 主控结论

这次纠偏之后，`v1.2` 的产品方向已经从“更大的工作台页”升级成：

> **主壳层中的顶层世界线工作区。**

这是一个更高层级、更像 Worldline 自己的产品定义。

下一步不应该直接进入重型实现，而是先把这个方向继续压成：

- 页面入口
- 路由关系
- 文件落点
- 状态层职责
- 最小验收门槛

然后再开始真正的前端实现。
