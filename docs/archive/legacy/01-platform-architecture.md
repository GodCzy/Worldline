# 01 平台架构

## 架构目标

世界线的平台架构目标是：在不破坏现有底座能力的前提下，为多主题兴趣知识场景建立一个清晰的双层结构。

## 双层结构

### 平台层

平台层负责所有跨主题共享能力，包括：

- 品牌
- 首页
- 主题分区
- 导航与路由
- 共享布局
- 权限与页面入口
- 主题上下文承接

### 模块层

模块层负责具体主题的领域能力，包括：

- 知识分类
- metadata schema
- 问答模板
- 图谱结构
- 推荐逻辑

## 当前仓库映射

当前物理目录仍保留上游运行结构，逻辑分层如下：

- `web/`：前端展示层与平台入口
- `server/`：路由与 API 入口
- `src/`：核心底座与运行能力
- `data/poe/`：PoE 模块样本数据
- `docs/`：世界线正式文档体系

## 平台层设计原则

1. 平台层不绑定某一个主题。
2. 平台层只负责承接上下文，不直接实现主题业务知识。
3. 平台层修改优先集中在首页、导航、主题页、路由、store 和低风险展示页。

## 模块层设计原则

1. 模块层拥有自己的知识组织方式。
2. 模块层可以定义自己的 metadata、图谱与推荐策略。
3. 模块层应通过配置和数据接入平台，而不是侵入底座核心。

## 主题上下文机制

当前世界线采用轻量主题上下文对象承接用户所在主题：

```json
{
  "theme": "poe",
  "module": "poe",
  "scene": "overview",
  "version": "phase2"
}
```

在实际页面联动中，还会扩展：

- `candidate`
- `graph`
- `build`
- `focus`
- `entry`

## 当前落地点

当前平台层落地点主要包括：

- 主页与主题入口
- `/themes/poe`
- `/agent`
- `/graph`
- `themeContext` 前端 store

## 高风险核心层

以下区域当前仍视为底座核心，不建议毕业设计阶段大改：

- `src/knowledge/**`
- `src/agents/common/**`
- `src/services/chat_stream_service.py`
- `src/services/agent_run_service.py`

## 当前完成度判断

平台层已经完成“可演示、可答辩”的最小闭环，但仍未完成“完全产品化”：

- 已完成：平台入口、主题页、主题上下文联动
- 未完成：更深层的主题上下文下沉到后台通用查询链
